    Test Planning

    Frontend Unit Test Cases

    Function: Login (Frontend)

    • If the user enters a valid username and password and clicks on the “Login” button, then they should be successfully redirected to the dashboard page and see their reimbursement requests.
    • If the login page is loaded properly, all fields (username, password) and buttons should be visible and functional.
    • If the user enters an incorrect username or password and clicks “Login,” an error message such as “Invalid username or password” should be displayed, and the page should not redirect.
    • If the user leaves the username or password field blank and clicks “Login,” an error message saying “Enter username and password to log in” should appear, and the login action should not be triggered.
    • If the user attempts to bypass the login page by directly visiting a protected route, they should be redirected back to the login page.

    Function: Logout

    • If the user clicks the “Logout” button, then their JWT tokens should be removed from local storage, and they should be redirected to the login page.
    • After logout, the user should not be able to access any protected routes or dashboard pages.
    • The logout button should be visible only when the user is logged in.
    • If the user uses the browser’s back button after logging out, they should still be redirected to the login page and not see cached private data.

    Function: User Dashboard (Employee)

    • If a logged-in employee visits their dashboard, then their submitted reimbursement requests should be listed correctly.
    • Each request should display relevant information such as title, amount, and status.
    • If no requests exist, the page should show a message like “No reimbursement requests yet. Click ‘Add Request’ to create one.”
    • Clicking a request should open a detailed view showing all associated expenses.
    • If the employee session has expired, they should be redirected to the login page.
    • If a user with the employee role tries to access the admin dashboard URL, they should see “Unauthorized” or be redirected to their own dashboard.

    Function: Add Reimbursement Request

    • If the user clicks “Add Request,” a form should appear with input fields for request details.
    • If all required fields are filled correctly and the user clicks “Create,” the request should be created successfully.
    • After creation, the user should see the new request in their dashboard list.
    • If any required field (like title) is empty, then validation messages should appear and prevent form submission.
    • If the user enters a very long title or description exceeding character limits, an error should appear like “Title too long.”

    Function: Add Expense to Reimbursement

    • If the user clicks “Add Expense,” a new expense form should appear with fields for title, amount, date, etc.
    • If valid data is entered and “Save” is clicked, the expense should appear in the list below.
    • If the user leaves required fields (like title or amount) empty, validation errors should appear.
    • If a non-numeric or negative value is entered for amount, an error “Amount must be greater than 0” should appear.

    Function: Request Approval (Admin)

    • If an admin logs in, they should see all reimbursement requests submitted by employees.
    • If the admin clicks on a specific request, they should see all details including employee name and expenses.
    • If the admin clicks “Approve”, the request status should change to “Approved”.
    • If the admin clicks “Reject”, the request status should change to “Rejected”.
    • If an employee tries to access the admin approval page, they should see “Unauthorized” access message.

    Backend Unit Test Cases

    Function: User Authentication

    • If the user sends a POST request to /api/auth/login/ with a valid username and password, then the server should respond with status 200 OK and return both access and refresh tokens.
    • If the credentials match a registered user, then the system should also include user role information in the response (admin/employee).
    • If incorrect credentials are sent, the API should return 401 Unauthorized with an error message “Invalid username or password.”
    • If either the username or password is missing in the request body, the server should return 400 Bad Request with a validation error.

    Function: User Management (Admin Only)

    • If an admin sends a valid POST request to /api/users/ with correct data, a new user record should be created, returning 201 Created.
    • If an admin sends a GET request to /api/users/, all users should be listed successfully.
    • If an admin sends a PATCH request to /api/users/{id}/ with valid fields, the user should be updated successfully.
    • If a non-admin user tries to access /api/users/, the response should be 403 Forbidden.
    • If required fields (like username or email) are missing, the API should return 400 Bad Request.
    • If a duplicate username or email is provided, an error “User with this email already exists” should be returned.

    Function: Reimbursement Request Creation

    • If an authenticated employee submits valid data via /api/requests/, the request should be created successfully with 201 Created.
    • The created request should be linked to the currently authenticated user.
    • If the title field is missing, the response should be 400 Bad Request with validation errors.
    • If invalid data types (e.g., string in amount) are sent, validation errors should appear.

    Function: Reimbursement Request Retrieval

    • If an authenticated employee sends a GET request to /api/requests/, they should receive only their own reimbursement requests.
    • If an admin sends the same request, they should receive all reimbursement requests.
    • If an unauthenticated user sends the request, it should return 401 Unauthorized.
    • If there are no reimbursement requests, the API should return an empty list with 200 OK.

    Function: Expense Management (Inside a Request)

    • If the user submits a valid POST request to /api/expenses/ with title, amount, and request ID, then the expense should be created successfully.
    • The expense should link to the correct reimbursement request.
    • GET /api/expenses/?request=\<id> should list all expenses belonging to that request.
    • If the linked reimbursement request ID does not exist, the API should return 400 Bad Request.
    • If the amount is negative or not numeric, the system should return a validation error.
